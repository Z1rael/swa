== Cross-cutting Concepts


ifdef::arc42help[]
[role="arc42help"]
****
.Content
This section describes crosscutting concepts (practices, patterns, regulations or solution ideas).
Such concepts are often related to multiple building blocks. 
They may include many different topics, such as the topics shown in the following diagram:

image::08-concepts-EN.drawio.png["Possible topics for crosscutting concepts"]

.Motivation
Concepts form the basis for _conceptual integrity_ (consistency, homogeneity) of the architecture. 
Thus, they are an important contribution to achieve inner qualities of your system.

This is the place in the template that we provided for a cohesive specification of such concepts.

Many of these concepts relate to or influence several of your building blocks. 

.Form
The form can be varied:

* concept papers with any kind of structure
* example implementations,especially for technical concepts
* cross-cutting model excerpts or scenarios using notations of the architecture views

.Structure
Pick **only** the most-needed topics for your system and assign each a level-2 heading in this section (e.g. 8.1, 8.2 etc).

DO NOT ATTEMPT to cover all of the topics of the aforementioned diagram.


.Further Information

Some topics within systems often concern multiple building blocks, hardware elements or development processes.
It might be easier to communicate or document such _cross-cutting_ topics at a central location, instead of repeating them in the description of the concerned building blocks, hardware elements or development processes.

Certain concepts might concern **all** elements of a system, others might only be relevant for a few.
In the diagram above, logging concerns all three components, whereas security is relevant only for two components.


See https://docs.arc42.org/section-8/[Concepts] in the arc42 documentation.

[#CCC_1]
=== CCC_1 Development - Build-Containers

Every microservice needs to be buildable via containers, to enhance CD-pipelines and make swapping dev-environments easier.

Every repo should contain a docker-compose file and additional Dockerfiles (named <Module>.Dockerfile, if there are multiple) as needed.

If multiple local environments are required, these should be implemented via docker-compose *profiles* and documented in the *README* of the project.

[#CCC_2]
=== CCC_2 Authentication & Authorization - External identity provider & JWT Token

For the autentication an external identity provider will be used.

If there is an issue with the authentication the user should be redirected to the login page of this identity provider, so that a login can be tried.

The identity provider sets a HttpOnly, Secure, Cookie containing a JWT-Token which includes the user and his various permissions.

These permissions are then to be used in the applications for the purpose of authorization.

[#CCC_3]
=== CCC_3 Communication and Integration - Message Brokers

Because of the microservices architecture, a strong tool for the communication between the various services is required.

For this Apache Kafka is to be used as an inbetween.

Internal requests between services and also request from the user clients via the API Gateway should pass through the message broker to allow for strong retryability and also upgradeability of the various services.

Exceptions can be made for idempotent request (especially GET), but these connections should be specifically documented.
